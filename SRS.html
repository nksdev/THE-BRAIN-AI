Software Requirements Specification (SRS) for Multi-Document Chatbot

1. Introduction

1.1. Purpose

This document specifies the comprehensive requirements for a Multi-Document Chatbot application, featuring a "Verified Brainmemory." The application is designed to enable users to upload documents, process them into a structured knowledge base, and engage in a conversational interface with the stored information. This SRS serves as a foundational reference for the development, testing, and maintenance of the software, ensuring all stakeholders have a shared understanding of the system's objectives and functionality.

1.2. Scope

The application is a web-based tool developed in Python using the Streamlit framework. It supports various document formats, including PDF, DOCX, TXT, and common image types (PNG, JPG, JPEG). The system leverages locally hosted Large Language Models (LLMs) via the Ollama platform for intelligent processing, summarization, and query-answering. The scope encompasses parallel document processing, a chat interface, a transparent knowledge base viewer, and a reporting utility. The focus is on a local-first, privacy-conscious design that provides a powerful alternative to cloud-based solutions.

1.3. Glossary

This section provides definitions for key terms, acronyms, and abbreviations used throughout this document.

  • SRS: Software Requirements Specification. A document that describes the system's functional and non-functional requirements.
  • LLM: Large Language Model. A type of artificial intelligence trained on vast amounts of text data to understand and generate human-like text.
  • UI: User Interface. The graphical layout and elements through which a user interacts with the application.
  • Ollama: An open-source framework for running large language models locally on a machine, eliminating the need for cloud-based API calls.
  • Brainmemory: The application's core knowledge base. Unlike a traditional vector database, it is stored in a human-readable, structured format (JSON and CSV) for transparency and verifiability.
  • RAG: Retrieval-Augmented Generation. A technique used to enhance LLM responses by first retrieving relevant information from a knowledge base and then using it to "augment" the prompt, grounding the answer in specific data.
  • TF-IDF: Term Frequency-Inverse Document Frequency. A statistical measure used to evaluate how important a word is to a document in a collection or corpus.

2. Overall Description

2.1. Product Perspective

The Multi-Document Chatbot is a standalone, web-based application. Its core functionality is self-contained and does not rely on external cloud services, as it utilizes a locally deployed LLM via the Ollama platform. The system manages its own data persistence by creating and maintaining a dedicated brainmemory directory for documents and knowledge patterns. This design prioritizes user privacy and independence from third-party services.

2.2. Product Functions

  • Document Management: The application provides a simple yet effective file uploader, allowing users to securely add documents of various formats (PDF, DOCX, TXT, images). These files are stored in a dedicated local directory, and a password-protected deletion feature ensures user control over their data.
  • Knowledge Processing: Users can choose between a fast, heuristic-based processing method and an intelligent, LLM-driven approach. The system extracts text from documents and processes it to generate structured knowledge patterns, including topics, entities, and summaries.
  • Conversational Interface: The core functionality is a chat interface that allows users to ask questions about the information contained in the documents. The system uses a RAG approach to provide answers that are accurate and grounded in the source material.
  • Knowledge Synthesis: A unique feature is the ability to generate a single, high-level knowledge pattern that synthesizes information across all stored documents, providing a unified view of the entire knowledge base.
  • Summarization and Reporting: The application can generate both page-specific summaries for PDF documents and a comprehensive "Consolidated Report" that brings together chat history and all summaries into a single, cohesive document for easy export.
  • Data Export: Features are provided to export the entire chat history, the structured knowledge base in CSV format, and any generated reports, giving users full control of their data.

2.3. User Characteristics

The target user base consists of individuals who need to analyze and interact with multiple documents, such as researchers, students, and professionals in fields requiring document-intensive work. Users are expected to have a basic understanding of a command-line interface to run the Python application and a local Ollama server.

2.4. Constraints

  • Platform: The application is restricted to a local machine with a Python 3.x environment.
  • LLM Dependency: Requires a local Ollama server for all LLM-related operations, as it is not designed to use external APIs.
  • Security: The deletion of all stored data is protected by a hardcoded password to prevent accidental data loss.
  • UI: The user interface is limited to the capabilities and widgets provided by the Streamlit framework.
  • Data Storage: All documents and knowledge patterns must be stored locally in the designated brainmemory directory.

2.5. Assumptions and Dependencies

  • It is assumed that the user has a functioning installation of Ollama with at least one pre-downloaded model.
  • The user is presumed to have adequate local storage to accommodate uploaded documents and the generated knowledge base.
  • All Python dependencies must be correctly installed and configured.

2.6. Comparison to Current State-of-the-Art and Project Uniqueness

This project stands apart from modern solutions by prioritizing a local-first, privacy-conscious architecture and introducing the "Verified Brainmemory" concept. While many contemporary multi-document analysis and RAG systems rely on complex, opaque vector databases and cloud-based LLM integrations, this application provides a robust, transparent, and user-centric alternative.

The key differentiators include:

  • Local-First and Privacy-Conscious Design: The application operates entirely on a local machine, ensuring sensitive data never leaves the user's environment. This provides a significant advantage for users working with confidential information.
  • The "Verified Brainmemory" Concept: Rather than using an opaque vector database, the project creates a human-readable knowledge base in JSON and CSV formats. This transparency allows users to directly inspect and validate the system's extracted information, fostering greater trust and confidence in the results.
  • Hybrid Processing Flexibility: By offering both an LLM-driven "Intelligent" method and a faster, heuristic-based "Fast" method, the application caters to a wider range of user hardware capabilities and project requirements, balancing performance and analytical depth.
  • Integrated Reporting and Synthesis: The capability to generate a "Consolidated Report" that synthesizes information from both chat history and document summaries transforms the tool from a simple chatbot into a powerful research and reporting assistant.

3. Specific Requirements

3.1. Functional Requirements

3.1.1. Document Upload and Processing

  • FR-1: The system shall provide a file uploader that accepts multiple files of type PDF, DOCX, TXT, PNG, JPG, and JPEG.
  • FR-2: Upon file upload, the system shall store the files in the /brainmemory/documents directory.
  • FR-3: The system shall process uploaded documents concurrently to extract text and generate knowledge patterns.
  • FR-4: The system shall offer two distinct methods for knowledge pattern generation: a) Intelligent (LLM), which uses a local LLM to generate topics, entities, and summaries, and b) Fast (Hybrid Approach), which uses local NLP algorithms for rapid pattern generation.
  • FR-5: The generated knowledge patterns shall be stored in a structured JSON file (knowledge.json).
  • FR-6: The system shall also append the generated knowledge patterns to a CSV file (knowledge.csv) for easy export.

3.1.2. Conversational Interface

  • FR-7: The system shall feature a chat interface for user queries.
  • FR-8: The user shall be able to select a specific document for a focused chat or a "General" option to query the entire knowledge base.
  • FR-9: The system's responses shall be synthesized from relevant knowledge patterns and the document content using a RAG approach.
  • FR-10: Chat history for each document and the "General" chat shall be preserved in the user's session state.

3.2. External Interface Requirements

3.2.1. User Interfaces

  • UI-1: The application shall present a clean, intuitive user interface built with Streamlit.
  • UI-2: The UI shall be logically divided into sections using tabs for "Chat," "Knowledge Base," and "Summarize & Report."
  • UI-3: A sidebar shall house global controls for model selection, document upload, and memory management.
  • UI-4: The UI shall provide clear user feedback, including progress bars for long-running tasks and status messages for operation outcomes.

3.2.2. Software Interfaces

  • SI-1: The system shall interface with the Streamlit framework for web application development.
  • SI-2: Communication with the Ollama API shall be implemented to utilize local LLMs.
  • SI-3: The PyPDF2 library shall be used for PDF text extraction.
  • SI-4: The docx library shall be used for DOCX text extraction.
  • SI-5: Pillow and pytesseract libraries shall be used for image-based text extraction.
  • SI-6: Standard Python libraries shall be employed for file I/O, parallel processing, and data serialization.

3.3. Non-Functional Requirements

3.3.1. Performance

  • NFR-1: The application's UI must remain responsive during document processing through the use of concurrent execution.
  • NFR-2: The "Fast (Hybrid Approach)" for pattern generation is expected to complete within 5 seconds for a typical 10-page document.

3.3.2. Security

  • NFR-3: Data deletion must be secured by a hardcoded password.
  • NFR-4: All sensitive data (documents and knowledge patterns) must be stored locally and not transmitted over external networks.

3.3.3. Reliability

  • NFR-5: The system shall gracefully handle corrupted or malformed documents, logging an error and continuing processing other files.
  • NFR-6: The system shall utilize robust error handling to manage potential failures during LLM calls, file operations, and data parsing.

3.4. Implementation and Processing Workflow

3.4.1. High-Level Architecture and Component Integration

The application's architecture is a single-page web application built with Streamlit. The key components are:

  • Streamlit UI (app.py): The front-end managing user interaction and state.
  • Core Logic (app.py): Contains all functions for data extraction, processing, and LLM interaction.
  • Local File System (brainmemory/): The local data persistence layer.
  • Ollama LLM Server: The local generative AI service.
  • LangChain Integration: The interface between the Python logic and the Ollama API.

3.4.2. Workflow Block Diagrams

The following diagrams provide a detailed, step-by-step illustration of the primary data and processing flows within the application.

Workflow 1: New Document Upload and Query

1. User Uploads PDF Document

The user selects a PDF file via the Streamlit file uploader. The application immediately saves the file to the /brainmemory/documents directory. For example: uploaded_report.pdf.

2. Concurrent Text Extraction & Processing

The application spawns a new thread or process for uploaded_report.pdf to prevent the UI from freezing. This thread performs two key actions:

2a. Text Extraction

The PyPDF2 library is used to read the PDF and extract all raw text. The extracted text is then segmented into chunks or paragraphs for easier processing.

2b. Knowledge Pattern Generation

A prompt is constructed and sent to the local Ollama LLM, along with the extracted text. The prompt asks the LLM to identify and extract topics, key entities, and a concise summary from the text. This is a crucial step to create the "Brainmemory."

3. 'Brainmemory' Data Storage

The LLM's structured output (topics, entities, summary) is saved into a new entry in the knowledge.json file. This process is repeated for every uploaded document. An entry is also added to knowledge.csv for easy export and inspection.

4. User Submits Query

The user enters a question in the chat, for example: "What is the main finding of the report?" The system identifies the query and its intent.

5. Retrieval-Augmented Generation (RAG)

The system prepares a highly-contextualized prompt for the LLM. This is done by:

5a. Retrieval

The system searches the knowledge.json and the raw text from uploaded_report.pdf to find all sentences or paragraphs most relevant to "main finding."

5b. Augmentation

The retrieved, relevant snippets are combined with the user's original question to create a new, enriched prompt. For example: "Using the following context, '...[relevant text snippets]...', what is the main finding of the report?"

6. Local Ollama LLM Call

The final, augmented prompt is sent to the locally running Ollama LLM via its API. The LLM processes this enriched context and generates a response based on the provided information, ensuring the answer is accurate and grounded in the source document.

7. Display Final Response

The LLM's generated answer is formatted and displayed in the chat interface. The response is directly tied to the information found in the documents, providing a reliable and verifiable result.

Workflow 2: Querying with No New Documents

1. User Submits Query

The user enters a question into the chat, for example: "What were the key topics from all documents?" No new documents are uploaded in this scenario.

2. Accessing Existing 'Brainmemory'

The application checks the /brainmemory directory and finds that the knowledge.json file already contains structured information from previous sessions. It loads this existing knowledge base into memory.

3. Retrieval-Augmented Generation (RAG)

The system prepares the prompt for the LLM by:

Retrieval

The application searches the loaded knowledge.json to find all topics and summaries that match the user's query about "key topics."

Augmentation

The retrieved topics and summaries are combined with the user's question to create a prompt. For example: "Using this list of topics and summaries from multiple documents, '...[all topics and summaries]...', what are the overall key topics?"

4. Local Ollama LLM Call

The augmented prompt is sent to the local Ollama LLM. The LLM's task is to synthesize a cohesive answer based on the pre-existing knowledge patterns it has been given.

5. Display Final Response

The final, synthesized answer is displayed in the chat. The user receives a comprehensive response that is based on the entire body of knowledge, even without a new document being uploaded.

3.4.3. Technical Concepts and AI/Generative AI

This section explains the core technical concepts and their application in the project.

a. Retrieval-Augmented Generation (RAG): RAG is a technique that enhances an LLM's capabilities by first retrieving relevant information from an external knowledge source and then using that retrieved information to augment the LLM's response. In this project, the chat functionality is a classic implementation of RAG. This process is crucial for providing accurate, document-grounded answers and minimizing the risk of the LLM "hallucinating" facts.

b. LLM (Large Language Model): An LLM is a form of artificial intelligence capable of understanding and generating human-like text. Within this application, the LLM serves as the central cognitive engine for tasks such as structured knowledge pattern generation, summarization, and RAG-based query answering. The choice of a local LLM via Ollama is a fundamental design decision that ensures data privacy.

c. Ollama: Ollama is an open-source platform that simplifies the process of running large language models locally on a machine. This design choice provides several key benefits: Data Privacy (all data remains on the user's machine), Offline Capability (the application functions without an internet connection), and Cost-Effectiveness (it eliminates the need for external, pay-per-use API costs).

d. The "Verified Brainmemory" Concept: This is a distinguishing feature of the project. Instead of relying on an opaque vector database that stores numerical embeddings, the application creates a human-readable knowledge base in knowledge.json and knowledge.csv. This transparency allows users to directly inspect and validate the extracted information, building a level of trust in the system's output that is not possible with black-box solutions.

e. AI Gen AI: This term refers to the use of an AI system (the LLM) to generate structured output that, in turn, is used by another AI process. In this project, the LLM generates the structured knowledge.json file, which is then used by the RAG system to inform conversational responses. This illustrates a multi-stage AI pipeline within the application's core logic.
